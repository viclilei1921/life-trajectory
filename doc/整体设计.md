# 生命轨迹（Life Trajectory）—— 设计文档

最近时常会回忆人生。但是好多过去的想法和生活场景完全不记得了。于是我有一个想法：搭建一座属于自己的**人生档案馆**，记录我的生命轨迹，让零散的记录自动串联成完整的生命轨迹。将我的生活数据化、可视化，是一个非常迷人且具有挑战性的工程项目。

这个项目还有一个隐藏目标：**用「记录」这件事本身，倒逼我养成持续写作的习惯**。所以产品的第一原则永远是——降低记录的摩擦力，让「留下一个节点」这件事随时随地能在 10 秒内完成。

> 本文档的定位：既是产品设计，也是技术蓝图，能够直接指导第一版（MVP）的开发。为了让每一节都能落地，我们从**系统架构**、**核心理念**、**功能与交互**、**数据结构**、**媒体存储**、**API 设计**、**备份与同步**、**开发步骤**这几个维度来展开。

## 一、系统整体架构设计

考虑到这是一个需要长期维护、承载大量个人数据的项目，稳定性、性能、以及**数据的自主可控**至关重要。核心原则是 **Local-First（本地优先）**：数据首先属于我自己，存在我自己的设备上，离线可用，云端只作为可选的备份/同步层。

| 层次 | 技术选型 | 说明 |
| --- | --- | --- |
| 后端支撑 | **Rust**（`axum` 或 `tauri` command 直连） | 高性能、内存安全。逻辑层与 UI 解耦，方便未来独立成服务。 |
| 前端展现 | **Svelte / SvelteKit + CSS** | 极少样板代码，构建响应迅速、跨设备（桌面 / 平板 / 手机）适配的现代化 UI。 |
| 桌面客户端 | **Tauri 2.0** | 把 Svelte 前端打包成轻量级跨平台安装程序（Win / macOS / Linux），未来 Tauri 2.0 也支持移动端（iOS / Android）。 |
| 数据库 | **SQLite（默认） / PostgreSQL（可选）** | 见下方说明。 |
| 全文检索 | **SQLite FTS5**（或后续 Tantivy） | 让「过去某天写过的一句话」能被搜到，这是回溯的关键。 |

### 关于数据库的决策

- **第一版用 SQLite**：单文件、零运维、天然适合 Local-First 桌面应用，备份就是拷一个文件，和「媒体文件单独存放、可快速备份」的诉求完美契合。
- **抽象出数据访问层（Repository trait）**：所有 SQL 通过一层接口访问，未来需要多端实时同步时，可以平滑切换到 **PostgreSQL** 而不动上层逻辑。
- 使用 **`sqlx`**（编译期校验 SQL）或 **`SeaORM`** 作为访问库，并用 **migration 文件**管理表结构演进。

### 逻辑分层

```
┌─────────────────────────────────────────┐
│  Svelte UI（时间轴 / 地图 / 3D / 编辑器） │
└───────────────┬─────────────────────────┘
                │ Tauri command（类型安全的 IPC）
┌───────────────▼─────────────────────────┐
│  Rust 应用层：用例逻辑 / 校验 / 检索      │
├──────────────────────────────────────────┤
│  Repository 层（trait 抽象）              │
├──────────────────────────────────────────┤
│  SQLite（元数据）  +  文件系统（媒体文件）│
└──────────────────────────────────────────┘
```

## 二、核心理念：五条轴线

一条「记录」不应该只是一段文字，而是被挂载到多条坐标轴上的一个**节点**。同一个节点会同时出现在不同的轴上，这正是「零散信息自动归位、天然形成轨迹」的核心机制。

1. **时间轴（主轴）**：一切记录的骨架。每个节点都有「发生时间」，支持模糊时间（只知道年 / 月 / 某个季节）。
2. **空间轴（地理轨迹）**：记录发生的地点（经纬度 + 地名），可在地图上连成移动轨迹。
3. **事件轴（内容分层）**：通过分类（日常 / 里程碑 / 旅行 / 想法…）和标签对内容分层，让不同粒度的记录各归其位。
4. **状态轴（量化维度）**：把「感受」变成可比较的数据——心情、精力、健康、体重、天气等，用于绘制人生的「情绪 / 状态曲线」。
5. **思想 / 人物轴（关系维度）**：记录里出现的**人**、引用的**思想 / 金句**，让你能沿着「某个人」或「某个念头」重新串起相关的所有节点。

> 每条轴既是一种**录入维度**，也是一种**回溯视图**。数据结构的设计目标，就是让同一份数据能被这五种视图自由切换。

## 三、核心功能与交互设计

「记录生命」最大的痛点在于**坚持**。设计必须尽可能降低输入门槛，同时提升回顾时的视觉震撼力。

### 3.1 三大设计准则

1. **低门槛记录**：拒绝「每日必写」的打卡压力，支持 10 秒快速记录，也支持深度长文。**留下节点，比完美记录更重要。**
2. **结构化沉淀**：所有记录自带时间、地点、分类属性，零散信息自动归位到人生坐标系，天然形成轨迹。
3. **情感化回溯**：可视化不是炫技，而是让你能「看见」自己的人生走向，在回顾中获得情感价值与成长启发。

### 3.2 录入体验（Capture）

- **快速捕捉框**：全局快捷键 / 托盘图标唤起，一个输入框 + 一键保存，自动带上「此刻的时间和地点」。
- **渐进式细化**：先随手记下，之后可以补充分类、地点、心情、照片——**先捕捉，后整理**。
- **草稿箱**：没写完的自动存草稿，永不丢失。
- **每日 / 情境提醒**（可选、可关）：温和地推送「今天想留下点什么吗？」，而不是冷冰冰的打卡。

### 3.3 回溯体验（Review）

- **时间轴视图**：主视图，按天 / 月 / 年缩放，像滚动一条河流一样回看。
- **地图视图**：把带地点的节点标在地图上，连成足迹。
- **状态曲线**：把心情 / 精力等量化维度画成折线，看见人生的起伏。
- **「那年今日」/ 随机回顾**：每天推送去年 / 前年的今天，或随机翻出一条旧记录，制造重逢感。
- **搜索**：全文检索 + 按标签 / 人物 / 地点 / 时间范围过滤。

### 3.4 后续增强（Roadmap）

1. **无感数据录入（ASR + LLM）**：接入语音识别（ASR）。直接对着设备口述当天的见闻或情绪，识别出的文本交由大型语言模型（LLM）处理——自动分段、提取地点 / 人物 / 心情、生成标题与标签。
2. **3D 轨迹可视化**：生命轨迹不应只是枯燥的二维列表。将人生节点、重要照片、甚至高分辨率视频映射到 3D 空间中（如 `Three.js` / `Threlte`），通过拖拽、缩放来「漫游」自己的过去。
3. **年度报告**：自动生成「年度生命轨迹总结」（去过的地方、见过的人、心情分布、写下的字数）。
4. **数据导入**：从相册（含 EXIF 时间地点）、聊天记录、旧日记批量导入，为轨迹补全历史。

## 四、数据结构设计

> 数据是最大的财富，这个还是你自己的，更应该设计合理。后续也许会增加，但**核心骨架不会改变**。

设计原则：
- **一张核心表 `entries`（记录 / 节点）** 承载五条轴线的主字段；细分维度（人物、标签、媒体、目标）用**独立表 + 关联表**，保证扩展时不动主表。
- **软删除**（`deleted_at`）而非物理删除——人生档案不该轻易丢失。
- 时间字段一律存 **UTC**，另存原始时区，展示时再转换。
- 所有主键用 **UUID / ULID**（方便未来多端合并，不会主键冲突）。

### 4.1 实体关系概览（ERD）

```
                         ┌──────────┐
                         │  people  │ (人物)
                         └────┬─────┘
                              │ N:M
┌──────────┐   N:M   ┌────────▼─────────┐   N:M   ┌────────┐
│  tags    ├────────►│     entries      │◄────────┤ goals  │ (人生目标)
└──────────┘         │ (记录 / 节点核心) │         └────────┘
                     └───┬──────────┬───┘
                     1:N │          │ 1:1（可空）
                ┌────────▼───┐  ┌───▼────────┐
                │   media    │  │ locations  │ (地点)
                │ (媒体元数据)│  └────────────┘
                └────────────┘
```

### 4.2 核心表：`entries`（记录 / 节点）

| 字段 | 类型 | 说明 | 对应轴 |
| --- | --- | --- | --- |
| `id` | TEXT (UUID) | 主键 | — |
| `title` | TEXT NULL | 标题，可空（快速记录时无需） | 事件 |
| `content` | TEXT | 正文，Markdown 格式 | 事件 |
| `entry_type` | TEXT | 类型：`note`(日常) / `milestone`(里程碑) / `thought`(想法) / `trip`(旅行) / `person`(关于某人) 等 | 事件 |
| `happened_at` | INTEGER | 事件发生时间（UTC 时间戳） | 时间 |
| `happened_precision` | TEXT | 时间精度：`exact` / `day` / `month` / `year`（支持模糊时间） | 时间 |
| `timezone` | TEXT | 原始时区，如 `Asia/Shanghai` | 时间 |
| `location_id` | TEXT NULL | 关联地点 | 空间 |
| `mood` | INTEGER NULL | 心情评分（如 -2 ~ +2 或 1~5） | 状态 |
| `energy` | INTEGER NULL | 精力 / 状态评分 | 状态 |
| `weather` | TEXT NULL | 天气 | 状态 |
| `metrics` | TEXT (JSON) NULL | 可扩展的量化维度（体重、睡眠、步数…），存 JSON | 状态 |
| `is_favorite` | INTEGER (0/1) | 是否收藏 / 精选 | — |
| `created_at` | INTEGER | 记录创建时间（UTC） | — |
| `updated_at` | INTEGER | 最后修改时间（UTC） | — |
| `deleted_at` | INTEGER NULL | 软删除时间戳，NULL 表示未删除 | — |

> **状态轴的取舍**：固定几个高频维度（`mood`/`energy`/`weather`）做成独立列以便高效画曲线；其余不定项塞进 `metrics` JSON，兼顾结构化与灵活性。

### 4.3 `locations`（地点）— 空间轴

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | TEXT (UUID) | 主键 |
| `name` | TEXT | 地点名称（如「家」「XX 咖啡馆」） |
| `latitude` | REAL NULL | 纬度 |
| `longitude` | REAL NULL | 经度 |
| `address` | TEXT NULL | 详细地址 |
| `created_at` | INTEGER | 创建时间 |

### 4.4 `people`（人物）— 关系轴

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | TEXT (UUID) | 主键 |
| `name` | TEXT | 姓名 / 称呼 |
| `relation` | TEXT NULL | 关系（家人 / 朋友 / 同事…） |
| `avatar_media_id` | TEXT NULL | 头像（指向 media） |
| `note` | TEXT NULL | 备注 |
| `created_at` | INTEGER | 创建时间 |

关联表 `entry_people`：`entry_id` + `person_id`（复合主键）。

### 4.5 `tags`（标签）与 `entry_tags`

关于「标签是否有必要」——**有必要，且很关键**。分类（`entry_type`）是粗粒度、互斥的骨架；标签是细粒度、可多选的血肉，是未来做「按主题回溯 / 可视化聚类」的基础。

`tags` 表：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | TEXT (UUID) | 主键 |
| `name` | TEXT UNIQUE | 标签名 |
| `color` | TEXT NULL | 展示颜色 |

关联表 `entry_tags`：`entry_id` + `tag_id`（复合主键）。

### 4.6 `media`（媒体元数据）— 见第五章

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | TEXT (UUID) | 主键 |
| `entry_id` | TEXT | 所属记录 |
| `file_hash` | TEXT | 文件内容哈希（SHA-256），用于寻址与去重 |
| `file_name` | TEXT | 原始文件名 |
| `media_type` | TEXT | `image` / `video` / `audio` / `file` |
| `mime_type` | TEXT | 如 `image/jpeg` |
| `size_bytes` | INTEGER | 文件大小 |
| `width` / `height` | INTEGER NULL | 图片 / 视频尺寸 |
| `duration_ms` | INTEGER NULL | 音视频时长 |
| `taken_at` | INTEGER NULL | 拍摄时间（来自 EXIF），可回填 `happened_at` |
| `created_at` | INTEGER | 创建时间 |

### 4.7 `goals`（人生目标）— 补充需求

人生目标也纳入系统，并可与记录关联（记录某个目标的进展节点）。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | TEXT (UUID) | 主键 |
| `title` | TEXT | 目标标题 |
| `description` | TEXT NULL | 描述 |
| `status` | TEXT | `active` / `done` / `paused` / `abandoned` |
| `target_date` | INTEGER NULL | 期望完成时间 |
| `created_at` | INTEGER | 创建时间 |
| `completed_at` | INTEGER NULL | 完成时间 |

关联表 `entry_goals`：`entry_id` + `goal_id`（把一条记录标记为「某目标的进展」）。

### 4.8 建表 SQL（SQLite 版，第一版骨架）

```sql
CREATE TABLE entries (
  id                 TEXT PRIMARY KEY,
  title              TEXT,
  content            TEXT NOT NULL DEFAULT '',
  entry_type         TEXT NOT NULL DEFAULT 'note',
  happened_at        INTEGER NOT NULL,
  happened_precision TEXT NOT NULL DEFAULT 'exact',
  timezone           TEXT NOT NULL DEFAULT 'Asia/Shanghai',
  location_id        TEXT REFERENCES locations(id),
  mood               INTEGER,
  energy             INTEGER,
  weather            TEXT,
  metrics            TEXT,               -- JSON
  is_favorite        INTEGER NOT NULL DEFAULT 0,
  created_at         INTEGER NOT NULL,
  updated_at         INTEGER NOT NULL,
  deleted_at         INTEGER
);

CREATE TABLE locations (
  id         TEXT PRIMARY KEY,
  name       TEXT NOT NULL,
  latitude   REAL,
  longitude  REAL,
  address    TEXT,
  created_at INTEGER NOT NULL
);

CREATE TABLE people (
  id              TEXT PRIMARY KEY,
  name            TEXT NOT NULL,
  relation        TEXT,
  avatar_media_id TEXT,
  note            TEXT,
  created_at      INTEGER NOT NULL
);

CREATE TABLE tags (
  id    TEXT PRIMARY KEY,
  name  TEXT NOT NULL UNIQUE,
  color TEXT
);

CREATE TABLE media (
  id          TEXT PRIMARY KEY,
  entry_id    TEXT NOT NULL REFERENCES entries(id),
  file_hash   TEXT NOT NULL,
  file_name   TEXT NOT NULL,
  media_type  TEXT NOT NULL,
  mime_type   TEXT,
  size_bytes  INTEGER,
  width       INTEGER,
  height      INTEGER,
  duration_ms INTEGER,
  taken_at    INTEGER,
  created_at  INTEGER NOT NULL
);

CREATE TABLE goals (
  id           TEXT PRIMARY KEY,
  title        TEXT NOT NULL,
  description  TEXT,
  status       TEXT NOT NULL DEFAULT 'active',
  target_date  INTEGER,
  created_at   INTEGER NOT NULL,
  completed_at INTEGER
);

-- 关联表
CREATE TABLE entry_people (
  entry_id  TEXT NOT NULL REFERENCES entries(id),
  person_id TEXT NOT NULL REFERENCES people(id),
  PRIMARY KEY (entry_id, person_id)
);
CREATE TABLE entry_tags (
  entry_id TEXT NOT NULL REFERENCES entries(id),
  tag_id   TEXT NOT NULL REFERENCES tags(id),
  PRIMARY KEY (entry_id, tag_id)
);
CREATE TABLE entry_goals (
  entry_id TEXT NOT NULL REFERENCES entries(id),
  goal_id  TEXT NOT NULL REFERENCES goals(id),
  PRIMARY KEY (entry_id, goal_id)
);

-- 索引：时间轴、地点、软删除过滤
CREATE INDEX idx_entries_happened_at ON entries(happened_at);
CREATE INDEX idx_entries_location    ON entries(location_id);
CREATE INDEX idx_entries_deleted     ON entries(deleted_at);

-- 全文检索（FTS5），支持对标题与正文的搜索
CREATE VIRTUAL TABLE entries_fts USING fts5(
  title, content, content='entries', content_rowid='rowid'
);
```

## 五、媒体存储

核心原则：**数据结构里不放文件，文件单独存放；数据库和文件都要能被快速、可靠地备份。**

### 5.1 存储布局

- 采用 **内容寻址（Content-Addressed Storage）**：文件名 = 内容的 SHA-256 哈希，天然**去重**（同一张图只存一份）、天然**防篡改**。
- 目录用哈希前两位分桶，避免单目录文件过多：

```
生命轨迹数据/
├── life.db                 # SQLite 数据库（元数据）
├── media/                  # 原始媒体文件
│   ├── a1/
│   │   └── a1b2c3...e9.jpg  # 文件名即哈希
│   └── f4/
│       └── f4d5...77.mp4
└── thumbnails/             # 自动生成的缩略图 / 预览图
    └── a1/
        └── a1b2c3...e9.webp
```

- 数据库里的 `media.file_hash` 指向 `media/` 中的真实文件；两者解耦。

### 5.2 缩略图与预览

- 图片入库时生成 WebP 缩略图，视频抽首帧做封面——保证时间轴 / 3D 视图滚动流畅，不必加载原图。
- 原图 / 原视频仅在点开时按需加载。

### 5.3 备份策略

- **数据库**：SQLite 单文件，直接拷贝即可；建议用 `VACUUM INTO` 生成一致性快照，避免拷贝到写入中途的文件。
- **媒体文件**：因为是内容寻址（文件名 = 哈希），增量备份极其友好——已存在的哈希文件不会重复传输。
- **一键导出**：把 `life.db` + `media/` 打包成一个 `.zip`，即可完整迁移 / 归档整个人生档案。
- 建议定期同步到外部磁盘 / 私有云（如坚果云、WebDAV、S3 兼容存储）。

## 六、API / 命令设计

因为是 Tauri 应用，前后端通过 **Tauri command** 通信（本质是类型安全的 IPC，未来也可平滑替换为 HTTP REST）。核心接口示例：

| 命令 | 入参 | 说明 |
| --- | --- | --- |
| `create_entry` | `NewEntry` | 新建一条记录（快速记录时大部分字段可空） |
| `update_entry` | `id, patch` | 更新记录（补充地点 / 心情 / 标签等） |
| `delete_entry` | `id` | 软删除 |
| `get_entry` | `id` | 获取单条详情（含媒体、标签、人物） |
| `list_entries` | `TimeRange, filters, page` | 时间轴分页拉取 |
| `search_entries` | `query, filters` | 全文 + 过滤搜索 |
| `attach_media` | `entry_id, file_path` | 导入文件、算哈希、生成缩略图、写元数据 |
| `list_entries_geo` | `bbox / TimeRange` | 地图视图用，返回带经纬度的节点 |
| `get_state_series` | `metric, TimeRange` | 状态曲线（心情 / 精力…）时间序列 |
| `on_this_day` | `date` | 「那年今日」回顾 |
| `export_archive` | `dest_path` | 导出完整档案 zip |
| CRUD | — | `people` / `tags` / `goals` / `locations` 各自的增删改查 |

## 七、开发步骤（含 MVP 范围）

分阶段推进，**先跑通一条最小闭环：能记录、能看回、能备份**，再逐步加维度。

### 阶段 0 — 环境与骨架
- 初始化 Tauri 2.0 + SvelteKit 项目，Rust 后端接入 SQLite（`sqlx` + migration）。
- 建好第四章的核心表结构。

### 阶段 1 — MVP：记录与时间轴（最小闭环）
- **录入**：快速捕捉框（标题可空 + Markdown 正文 + 自动带时间）。
- **回看**：时间轴列表视图，按时间倒序，支持分页。
- **媒体**：支持给记录添加图片（内容寻址存储 + 缩略图）。
- **备份**：一键导出 zip。
- ✅ 达成目标：**从今天起，我可以每天往里面写东西了。**（写作习惯从这里开始）

### 阶段 2 — 结构化维度
